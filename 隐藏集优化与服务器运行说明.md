# 隐藏集优化与服务器运行说明

## 这次主要优化了什么

本版本把系统进一步向 PaSa 的 Crawler + Selector 思路靠拢，重点不是记公开测试集答案，而是提升隐藏查询上的召回稳定性。

1. 增加 Serper/arXiv 召回链路：系统可通过 Google/Serper 搜索 `site:arxiv.org/abs`，从网页结果中抽取 arXiv ID，再调用官方 arXiv API 回填论文标题、摘要、年份和作者。
2. 保留通用学术 API：OpenAlex、Semantic Scholar、arXiv 原生搜索仍然保留，用来覆盖非 arXiv 论文和普通学术查询。
3. 保留 PaSaTitleDB 作为本地高召回源：如果服务器存在 `data/pasa-dataset/paper_database/id2paper.json`，会自动启用；如果隐藏集不提供这个库，系统会自动跳过，不影响运行。
4. 评测模式平衡 F1 和召回：`evaluate_pasa.py` 默认把候选池扩大到至少 260，API 预算扩大到至少 64，LLM selector 验证数扩大到至少 80，并以 20 篇/批减少 DeepSeek 往返次数。
5. Selector 优先级调整：Serper/arXiv、arXiv、PaSaTitleDB 这类更接近 PaSa Crawler 的来源，会优先进入 LLM relevance verifier，减少被 OpenAlex 噪声挤掉的概率。
6. 运行效率优化：在线 API 检索支持并发和进程内缓存；Embedding/Reranker 对同一查询、同一论文的分数也会缓存，避免多轮检索反复算小模型。

## 为什么这对隐藏集更有用

隐藏集不能依赖公开答案，也不能只靠 PaSa 下载的本地标题库。更稳的策略是：

- 用 LLM/规则把自然语言问题拆成多个检索式；
- 用 Serper 搜索引擎先找到真实 arXiv 页面；
- 用 arXiv API 拿可靠元数据；
- 用 embedding、reranker、LLM selector 重新排序；
- 最后按权威性、时效性、相关性和多样性综合排序。

这比单纯 OpenAlex/Semantic Scholar 更接近 PaSa 的强 Crawler 方法，也比只用本地库更适合隐藏测试。

## 服务器需要配置的环境变量

DeepSeek 用于查询理解、候选验证和结果归纳：

```bash
export DEEPSEEK_API_KEY="你的DeepSeek Key"
```

Serper 用于 Google 风格搜索 arXiv 页面，建议正式评测时配置：

```bash
export SERPER_API_KEY="你的Serper Key"
```

Semantic Scholar Key 可选，不配置也能跑，只是可能遇到限流：

```bash
export SEMANTIC_SCHOLAR_API_KEY="你的Semantic Scholar Key"
```

注意：Key 只放在环境变量里，不要写进 Git 仓库和截图里。之前截图中出现过 DeepSeek Key，建议在平台上重置一次。

## 服务器正式评测命令

```bash
cd ~/wenyan-zhisou-advanced/competition
source .venv/bin/activate

export DEEPSEEK_API_KEY="你的DeepSeek Key"
export SERPER_API_KEY="你的Serper Key"

python evaluate_pasa.py \
  --config config.yaml \
  --input data/pasa-dataset/RealScholarQuery/test.jsonl \
  --output_dir runs/pasa_rs_hidden_style \
  --limit 5 \
  --top_k 20

cat runs/pasa_rs_hidden_style/metrics.json
```

如果想跑完整 PaSa RealScholarQuery 测试集，把 `--limit 5` 删除：

```bash
python evaluate_pasa.py \
  --config config.yaml \
  --input data/pasa-dataset/RealScholarQuery/test.jsonl \
  --output_dir runs/pasa_rs_full \
  --top_k 20
```

## 看结果时重点关注什么

1. `recall@100`：说明 Crawler 是否把正确论文召回进候选池。
2. `recall@20` 和 `precision@20`：说明 Selector 和排序是否把正确论文推到了前面。
3. `hit_report.json`：逐条看哪些 gold 没命中，是检索没召回，还是召回了但排序太低。
4. `LLM Calls`：如果一直是 0，说明 DeepSeek 没有真正参与 selector。
5. 网页左侧配置提示：正式运行时应看到 `Serper/arXiv`、`arXiv`、`OpenAlex`、`Semantic Scholar` 等检索源。
