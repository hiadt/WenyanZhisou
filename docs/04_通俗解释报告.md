# 通俗解释报告

这个项目可以理解成一个“会自己想搜索词、会多平台查论文、会排序、会解释结果的科研助手”。

普通搜索是你输入什么，它就搜什么。这个系统多做了几步：

1. 先理解你真正想找什么。
2. 把一个复杂问题拆成几个更适合搜索的小问题。
3. 去 OpenAlex 和 Semantic Scholar 这类学术数据库找论文。
4. 把重复论文合并掉。
5. 根据引用关系补充相关论文。
6. 用小模型给论文打分。
7. 用大模型检查前排论文是不是真的相关。
8. 最后按综合分排序，展示给你看。

## 为什么要大模型和小模型一起用

大模型聪明，但贵、慢，不适合把几百篇论文全都仔细看一遍。

小模型便宜、快，适合批量算相似度和排序，但它对复杂语义的理解不如大模型。

所以本项目的做法是：

- 大模型负责“想清楚要搜什么”和“检查重点论文”。
- 小模型负责“快速筛大量论文”。

这样既有智能性，又不会太慢太贵。

## 前端页面怎么看

搜索框：输入你的研究问题。

Top K：想返回多少篇论文。

搜索论文：真正执行搜索。

Smoke 评测：用内置小样例检查程序是否能跑。

RAG 评测：展示如何基于论文证据组织回答。

漏洞检测：检查当前配置和运行链路有没有明显问题。

论文排序：看最终推荐论文，右边分数越高越靠前。

查询拆解：看系统把你的问题拆成了哪些搜索词。

关系图：看论文之间的引用关系。

JSON：看程序原始输出，方便调试和提交。

## 为什么普通电脑跑不动

如果只是打开前端、调用在线 API，普通电脑可以跑。

但正式比赛如果要加载真实 embedding 模型、reranker 模型，甚至本地大模型，就会占用很多显存和内存。16GB 内存电脑不是完全不能跑，但很容易在安装 PyTorch、下载模型、加载模型时卡住或失败。

所以现在建议租服务器。

## 租什么服务器

先租 RTX 4090 24GB + 64GB 内存就够试正式流程。

如果要更稳，租 L20 48GB 或 A100 40GB。

如果你用 DeepSeek 这类在线大模型 API，就不用在服务器上本地跑大模型，服务器主要负责小模型排序和评测。

## 比赛时怎么讲

可以这样讲：

“我们的系统不是简单关键词搜索，而是一个学术检索 Agent。它先用大模型理解科研意图并拆解查询，再调用 OpenAlex 和 Semantic Scholar 获取候选论文，之后用 embedding 和 reranker 小模型做语义排序，最后用大模型对高分候选进行相关性验证。系统输出论文列表、查询拆解、引用关系图和评测指标，能够对齐赛题对 F1、效率和结构化结果的要求。”

新版答辩可以再补一句：

“我们参考 PaSa 的 Crawler/Selector 思想做了轻量化实现，但没有依赖 PaSa 的训练 checkpoint。Crawler 负责多策略搜索、查询演化和引文扩展，Selector 负责候选过滤和大模型相关性判断。排序时不仅看相关性，还加入权威性、时效性和多样性分数，前端还能展示 Agent 每一步搜索轨迹。”
