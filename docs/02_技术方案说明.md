# 技术方案说明

## 1. 总体流程

```text
query
  -> LLMPlanner / fallback planner
  -> CrawlerAgent: semantic / constraint / authority / recency strategies
  -> OpenAlex + Semantic Scholar + local corpus
  -> deduplicate
  -> LLMQueryEvolver / heuristic query evolution
  -> one-hop citation network expansion
  -> BM25 / Embedding / Reranker scoring
  -> SelectorAgent: batched LLM verifier and noise filtering
  -> authority / recency / diversity aware final ranking
  -> ResultSynthesizer
  -> web / json / evaluation output
```

## 2. 查询规划

`LLMPlanner` 位于 `competition/wenyan_competition/llm.py`。启用大模型时，它会输出：

- 原始查询
- 研究意图
- 关键实体
- 方法词
- 数据集词
- 约束条件
- 子查询列表
- 负向过滤词

禁用大模型时，fallback planner 会根据规则扩展英文关键词，保证系统仍可运行。

查询规划之后，系统还会根据第一轮候选论文做二轮查询演化。启用 LLM 时由 `LLMQueryEvolver` 生成 3-5 条新的英文检索式；LLM 不可用时，系统会从高分候选标题、摘要中抽取高频主题词，拼接成下一轮查询。这一步用于补强“迭代检索”，但最终是否提升指标仍要通过测试集验证。

本项目采用轻量 PaSa-inspired 设计：不复刻 PaSa 的 Crawler/Selector 训练和强化学习 checkpoint，而是在可部署约束下实现 Crawler/Selector 职责拆分。Crawler 负责多策略检索、查询演化和引文网络扩展；Selector 负责抽取候选队列、调用 LLM verifier、过滤低质量论文。每一步会写入 `agent_trace`，可在前端“Agent轨迹”查看。

## 3. 检索与候选池

`AcademicRetriever` 位于 `retrievers.py`。

检索源包括：

- OpenAlex：覆盖广，适合基础召回。
- Semantic Scholar：论文元数据质量较高，适合补充引用关系。
- LocalCorpus：用于 smoke 测试和私有语料扩展，支持读取 `full_text`、`fullText`、`text`、`body` 等全文字段。

候选合并后使用 DOI、paper id、标题归一化进行去重。

在线 OpenAlex / Semantic Scholar 通常只能稳定拿到标题、摘要、元数据和部分引用关系；本项目的“全文检索”主要面向本地私有语料或后续 PDF 解析结果。只要数据里写入全文字段，BM25、Embedding、Reranker 都会把全文前若干字符纳入打分。

Crawler 的检索策略包括：

- semantic-core：使用查询拆解得到的核心语义子查询，优先保证召回。
- constraint-focused：显式拼接方法、数据集、venue、时间等约束，减少宽泛噪声。
- authority-oriented：加入 survey、review、benchmark、influential 等词，补充权威/综述类论文。
- recency-oriented：当查询含“最新、近年、SOTA、年份”等信号时，偏向近期论文。
- citation-network expansion：从高分种子论文的一跳 references/citations 拉取补充候选。

## 4. 引用扩展

系统会从高分候选中选取若干篇作为种子，读取 references 和 citations，补充一跳关系论文。这样可以找回关键词不完全命中但与核心论文高度相关的文献。

相关参数：

- `citation_expand_seeds`
- `citation_expand_limit`

## 5. 小模型排序

`models.py` 包含两类小模型：

- `EmbeddingModel`：默认可加载 `BAAI/bge-small-en-v1.5`。
- `CrossEncoderReranker`：默认可加载 `BAAI/bge-reranker-base`。

如果启动时加 `--fallback_models`，系统不加载真实模型，改用稀疏向量和启发式 reranker，适合普通电脑和前端调试。

## 6. 综合排序

`ranker.py` 将多个信号归一化后加权融合。默认权重可在 `config.yaml` 修改：

```yaml
ranking:
  api_weight: 0.10
  bm25_weight: 0.15
  embedding_weight: 0.25
  reranker_weight: 0.25
  llm_verifier_weight: 0.15
  authority_weight: 0.06
  recency_weight: 0.03
  diversity_weight: 0.01
```

调参建议：

- 召回不足：增大 `per_query`、`max_candidates`、`citation_expand_limit`。
- 结果太杂：增大 `reranker_weight` 或启用 `llm_verifier_weight`。
- 结果过于同质：增大 `diversity_weight`。
- 希望更偏经典高影响论文：增大 `authority_weight`。
- 查询强调最新进展：增大 `recency_weight`，并提高 `max_candidates`。
- LLM 成本太高：降低 `llm_verify_top_n` 和 `max_llm_calls_per_query`。
- 显存不足：降低 `embedding_batch_size` 和 `reranker_batch_size`。

## 7. 评测

`evaluate_pasa.py` 读取 JSONL 测试集，按查询逐条检索，并输出指标。

支持指标：

- Precision@20
- Recall@20
- Recall@50
- Recall@100
- F1@20
- 平均 API calls
- 平均 LLM calls
- 平均 latency

## 8. 前端

`web_demo.py` 使用 Python 标准库 HTTP server 实现，不依赖额外前端框架。这样服务器部署更简单，租 GPU 机器后可以直接启动。

前端与后端共用同一套 `AcademicSearchAgent`，因此演示结果和命令行评测逻辑一致。

前端页面包含：

- 论文排序：展示最终排序、标题、年份、来源、摘要和各路分数。
- 查询拆解：展示意图、实体和子查询。
- Agent轨迹：展示 Planner、Crawler、Selector、Ranker 每一步执行了什么、用了哪些查询、候选数量如何变化。
- 结果归纳：展示整体结论、主题线索、高度相关候选、部分相关候选、证据缺口和下一轮检索建议。
- 关系图：展示候选论文之间的引用/参考关系。
- JSON：展示完整 API 返回，便于调试和写报告。
