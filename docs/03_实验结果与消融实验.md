# 实验结果与消融实验

## 1. 实验目的

当前仓库内置的是可复现 smoke 实验，用于验证工程链路是否完整。它不是最终比赛成绩。正式成绩需要在官方公开测试集或隐藏测试集上运行。

## 2. Smoke 实验

运行：

```bash
cd competition
python evaluate_pasa.py --config config.smoke.json --input data/sample_eval.jsonl --output_dir runs/smoke_eval --no_llm --fallback_models --top_k 20
```

输出：

```text
runs/smoke_eval/
├── predictions.jsonl
├── metrics.json
└── report.md
```

该实验验证：

- 查询规划是否可用。
- 本地语料检索是否可用。
- 候选去重是否可用。
- BM25 / fallback embedding / fallback reranker 是否可用。
- 指标计算和结果落盘是否可用。

## 3. 在线检索实验

运行：

```bash
python run_agent.py --config config.online.json --query "intelligent vehicle cybersecurity risk assessment" --output runs/online_vehicle_security.json --no_llm --fallback_models --top_k 10
```

该实验验证：

- OpenAlex 是否可以访问。
- Semantic Scholar 是否可以访问。
- arXiv / Serper 是否可以访问。
- 中文或英文查询能否被扩展成有效英文检索词。
- 搜索结果是否随查询变化。

## 4. 正式评测实验

租服务器后使用：

```bash
python evaluate_pasa.py --config config.yaml --input data/RealScholarQuery/test.jsonl --output_dir runs/pasa_realscholar
```

如果先小规模试跑：

```bash
python evaluate_pasa.py --config config.yaml --input data/RealScholarQuery/test.jsonl --output_dir runs/pasa_realscholar_50 --limit 50
```

调参阶段建议先用：

```bash
python evaluate_pasa.py --config config.yaml --input data/RealScholarQuery/test.jsonl --output_dir runs/pasa_limit5 --limit 5
```

最终冲分再用：

```bash
python evaluate_pasa.py --config config.yaml --input data/RealScholarQuery/test.jsonl --output_dir runs/pasa_all
```

## 5. 消融实验设计

建议正式报告中按以下设置做消融：

| 设置 | 说明 |
|---|---|
| BM25 only | 只使用关键词匹配和 API 分 |
| + Query expansion | 增加查询拆解和子查询扩展 |
| + Citation expansion | 增加引用/被引扩展 |
| + Embedding | 增加语义向量分数 |
| + Reranker | 增加 cross-encoder 精排 |
| + LLM verifier | 增加大模型相关性验证 |
| + Serper/arXiv Crawler | 增加搜索引擎到 arXiv 页面召回 |
| + Crawler/Selector trace | 增加轻量 PaSa-inspired 多策略检索和候选筛选轨迹 |
| + Authority/Recency/Diversity | 增加权威性、时效性、多样性融合排序 |
| + Cache/Parallel | 增加 API 并发、API 缓存和小模型分数缓存 |

每组记录：

- F1@20
- Recall@50
- Recall@100
- 平均延迟
- 平均 API 调用次数
- 平均 LLM 调用次数

## 6. 预期现象

- Query expansion 通常提升召回率。
- Citation expansion 能补充关键词不明显但学术关系密切的论文。
- Embedding 对跨表达、同义词、长查询更有帮助。
- Reranker 提升前排结果质量。
- LLM verifier 提升复杂意图下的相关性判断，但会增加成本和延迟。
- Crawler/Selector 轨迹可以证明系统不是一次性关键词搜索，而是有多策略召回、候选筛选和可解释搜索过程。
- Authority/Recency/Diversity 分数用于缓解只看相关性导致的结果同质化、低质量论文靠前等问题。
- Serper/arXiv 通常会提高 arXiv 类查询召回，但如果排序没跟上，可能出现 Recall@100 提升而 F1@20 不明显的情况。
- Cache/Parallel 不改变排序逻辑，理论上主要降低平均延迟；如果接口限流严重，可降低 `api_parallelism`。

## 7. F1@20 诊断方法

F1@20 同时受 Precision@20 和 Recall@20 影响。正式评测后重点看 `hit_report.json`：

- gold 论文在前 100 都没出现：主要是召回问题，应检查 Serper/arXiv、arXiv、OpenAlex、Semantic Scholar 和 PaSaTitleDB 是否启用。
- gold 论文在 50/100 内出现但没进前 20：主要是排序问题，应提高 `llm_verify_top_n`、`llm_verifier_weight`，或检查 LLM verifier 是否真正调用。
- 前 20 结果很多相似但不命中：需要增加多样性，或降低过强的 authority/recency 偏置。
- 延迟过高：先保持 `enable_api_cache: true`，再调整 `api_parallelism`、`llm_verifier_batch_size` 和 `max_api_calls_per_query`。
- 调参很慢：先用 `--limit 5` 或 `--limit 20` 小样本验证；指标确认后删除 `--limit` 跑完整测试集。

## 8. 注意事项

Smoke 结果不能写成最终比赛成绩。正式报告里应明确区分：

- 工程 smoke 实验：证明代码可运行。
- 公开测试集实验：证明方法有效。
- 隐藏测试集：以官方平台结果为准。
