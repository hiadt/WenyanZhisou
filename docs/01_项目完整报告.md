# 问研智搜项目完整报告

## 1. 项目定位

本项目面向科研论文检索与推荐场景，目标是解决“查询表达复杂、关键词不稳定、检索结果多且杂、人工筛论文耗时”的问题。系统将用户输入的自然语言研究问题转化为结构化检索计划，调用外部学术 API 获取候选论文，再结合大小模型协同排序，最终输出论文列表、查询拆解、关系图和评测结果。

项目当前只保留 Python 竞赛版，主要服务于赛题三的正式评测和答辩演示。

## 2. 行业背景

科研检索传统上依赖关键词匹配。对于简单主题，这种方式足够；但面对跨领域、长问题、含方法和约束条件的查询时，关键词检索容易漏掉语义相关论文，也容易返回主题相近但实际无关的论文。

近年来学术搜索系统逐渐从“关键词检索”走向“语义检索 + 大模型理解 + 多源融合”。典型方向包括：

- 使用大模型理解研究意图并改写查询。
- 使用 embedding 模型计算查询与论文摘要的语义相似度。
- 使用 reranker 对候选结果二次精排。
- 使用引用关系补充关键词不容易命中的重要论文。
- 使用结构化指标评估召回率、F1、延迟和调用成本。

## 3. 系统架构

系统由五层组成：

```text
用户问题
  ↓
LLMPlanner：查询理解、实体抽取、子查询拆解
  ↓
AcademicRetriever：OpenAlex / Semantic Scholar / 本地语料检索
  ↓
AcademicSearchAgent：候选合并、去重、引用扩展、多轮检索
  ↓
CompetitionRanker：BM25 + Embedding + Reranker + LLM verifier 综合排序
  ↓
论文列表 / JSON / 关系图 / 评测指标 / 前端展示
```

核心源码位于：

```text
competition/wenyan_competition/
├── agent.py
├── config.py
├── dataset.py
├── llm.py
├── metrics.py
├── models.py
├── ranker.py
├── retrievers.py
└── schema.py
```

## 4. 大小模型协同

## 3.1 针对参考系统挑战的改进

赛题提到 PaSa、SPAR、Ai2 Paper Finder 等系统已经证明 LLM Agent 在学术搜索中的价值，但仍存在查询理解不足、召回与精度难平衡、权威性/时效性/多样性难统一、结果归纳展示不足等问题。本项目当前版本不是直接复刻 PaSa，而是实现了轻量 PaSa-inspired Crawler/Selector 架构：

- Crawler：负责 semantic-core、constraint-focused、authority-oriented、recency-oriented 多策略召回，并根据高分候选执行查询演化和一跳引文网络扩展。
- Selector：负责从候选池中抽取高分与多样候选，分批调用 LLM verifier 判断高相关、部分相关或不相关，降低噪声。
- Ranker：融合 API、BM25、Embedding、Reranker、LLM verifier、authority、recency、diversity 八类信号，显式平衡相关性、权威性、时效性和多样性。
- Web Demo：新增 Agent 轨迹页签，展示 Planner、Crawler、Selector、Ranker 每一步做了什么，便于答辩解释系统不是一次性关键词搜索。

这部分是本项目相对最初轻量版本的主要增强点。它仍然比 PaSa 的强化学习训练系统轻量，但更适合当前服务器部署、网页演示和比赛工程复现。

## 4. 大小模型协同

大模型负责“理解和判断”，小模型负责“批量计算和排序”。

大模型模块：
- 查询拆解：把复杂问题拆成多个子查询。
- 检索策略：生成英文扩展词、方法词、同义词。
- 相关性验证：对 Top N 候选进行语义判断。

小模型模块：
- Embedding：计算查询与论文文本的语义相似度。
- Reranker：对候选论文做精排。
- Fallback 模型：在没有 GPU 或没有模型权重时，用 BM25 和启发式方法保证系统可运行。

这种设计的好处是：大模型调用次数少，主要处理高价值判断；小模型处理大量候选，降低成本和延迟。

## 5. 检索来源

系统已接入：

- OpenAlex Works API
- Semantic Scholar Graph API
- 本地 JSONL 语料

正式比赛推荐使用 OpenAlex + Semantic Scholar 双源检索。本地 JSONL 只用于离线 smoke 测试或后续扩展私有论文库。

## 6. 排序策略

每篇论文会得到多个分数：

- `api_score`：外部 API 初始排序信号。
- `bm25_score`：关键词匹配程度。
- `embedding_score`：语义相似度。
- `reranker_score`：精排模型分数。
- `llm_score`：大模型相关性验证分数。

最终分数为加权融合：

```text
final_score =
  api_weight * api_score
  + bm25_weight * bm25_score
  + embedding_weight * embedding_score
  + reranker_weight * reranker_score
  + llm_verifier_weight * llm_score
```

权重在 `config.yaml` 的 `ranking` 中调整。

## 7. 前端演示

前端由 `web_demo.py` 提供，直接调用同一套 Python 竞赛管线，不再保留单独的旧演示版本。

页面包含：

- 搜索框：输入研究问题。
- Top K：控制返回论文数量。
- 搜索论文：调用真实检索与排序流程。
- Smoke 评测：跑本地样例评测。
- RAG 评测：展示结构化回答与证据。
- 漏洞检测：检测配置和运行链路风险。
- 论文排序：展示每篇论文及各项分数。
- 查询拆解：展示大模型或 fallback planner 的查询计划。
- 关系图：展示论文引用/被引关系。
- JSON：展示原始结构化输出。

## 8. 评测方式

评测脚本是：

```text
competition/evaluate_pasa.py
```

输入为 JSONL，每行包含查询和标准相关论文 ID。输出包括：

- `predictions.jsonl`
- `metrics.json`
- `report.md`

指标包括：

- Precision@20
- Recall@20
- Recall@50
- Recall@100
- F1@20
- API 调用次数
- LLM 调用次数
- 端到端延迟

这些指标对应赛题中“F1 Score、运行效率、结构化输出”的要求。

## 9. 硬件与部署

如果只跑在线 API + fallback 小模型，普通电脑即可。正式比赛建议租 GPU 服务器，因为真实 embedding/reranker 和可能的本地 LLM 都需要显存。

推荐：

- 调试：8 vCPU、32GB RAM、无 GPU 或 24GB GPU。
- 正式：8-16 vCPU、64GB RAM、RTX 4090 24GB / A10 24GB / L20 48GB / A100 40GB+。
- 如果本地部署 7B LLM，优先 L20 48GB 或 A100 40GB+。

## 10. 项目价值

本项目不是单纯网页搜索，而是一个面向科研检索的 Agent 系统。它把“理解问题、生成检索策略、调用学术 API、候选合并、语义排序、结构化输出、指标评测”串成完整链路，能够直接用于比赛验证和答辩展示。
